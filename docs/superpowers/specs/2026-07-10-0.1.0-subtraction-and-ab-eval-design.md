# 0.1.0 — Subtraction release, gated by an A/B eval

**Date:** 2026-07-10
**Status:** Approved design (brainstorm 2026-07-10)
**Origin:** Whole-project complexity review (2026-07-10), thesis "is ultrapowers too complex?"

## Goal

Ship 0.1.0 as a **subtraction-only** release that removes the ballast the complexity
review identified, demotes sealed acceptance to opt-in, and freezes the verification
periphery — then prove, before tagging, that the cuts cost nothing on the three value
propositions (code quality, token economy, clock speed) via a low-volume A/B eval
run with `claude -p`.

Two principles bind every task in this effort:

1. **No new machinery in the release.** 0.1.0 deletes and demotes; it adds only the
   eval kit, which lives in `evals/` and never runs in a user's session.
2. **The eval is the tag gate.** We do not tag 0.1.0 until the A/B numbers meet the
   tag rule written below — decided now, before we see any numbers.

## Background (evidence, condensed)

The 2026-07-10 review found the value loop (waves.js engine, review discipline,
compiler core, suite-gate, git trust checks) sound and field-proven: ~60 tasks across
13 harvested runs, zero implementer-caused redirect rounds. The instability lives in
the verification periphery: releases 0.0.32–0.0.36 were the apparatus repairing
itself; 6 of 10 field failures at 0.0.35 were the sealing machinery's own
false-blocks. Roughly 1,000 of `compile_plan.py`'s 1,647 lines are diagnostics and
guards against its own past over-inference; ~25 test files (~100 tests) are
substring-in-markdown prose pins that survive semantic regressions.

## Part 1 — The subtraction list

### 1a. Delete prose-pin and self-policing tests

**Definite deletions (11 files):** `test_cross_phase_review.py`,
`test_literal_trace_selfreview.py`, `test_sibling_by_role_rule.py`,
`test_finishing_notes.py`, `test_salvage_gate.py`, `test_reading_lenses.py`,
`test_viewer_offer_touchpoints.py`, `test_review_trigger_prose.py`,
`test_compat_skill_wiring.py`, `test_complexity_metric.py`,
`test_complexity_ratchet.py` (plus any script that exists only to serve the
ratchet/metric).

**Definite keeps (4 files):** `test_no_prompt_drift.py` (byte-equality of baked
prompts), `test_recommendation_rubric.py` (pins branch order),
`test_marker_contract.py`, `test_version_sync.py` (the plugin.json/marketplace.json
parity gotcha is real and has bitten).

**Rule for the remaining prose-adjacent files** (`test_ultraplan_skill.py`,
`test_ultradocket_skill.py`, `test_ultralearn_skill.py`, `test_orchestrator_markers.py`,
`test_report_runbook.py`, `test_path_variables.py`, and similar): delete each test
function that only asserts a phrase appears somewhere in a document; keep tests that
pin order, byte-equality against a source file, or a shipped runtime contract. A file
left empty by this rule is deleted.

### 1b. Trim compiler diagnostics

In `compile_plan.py`:

- Keep the two `marker_conflicts` kinds SKILL.md renders (`conflict`, `inference`).
  Delete the near-miss-spelling surfacings and other soft-conflict categories that
  feed neither bucket nor any hard refusal.
- Delete wave-label derivation (`_title_words`, `_shared_title_noun`,
  `_common_file_dir`, `derive_wave_label`, and the `waveLabels` emission). The engine
  already falls back to the single-task title or `Wave N`; progress labels get
  plainer, nothing else changes. `waves.js` is NOT modified — it already handles
  absent `waveLabels`.
- Every hard refusal stays: cycles, duplicate ids, malformed headings, no tasks,
  Files grammar, fence-awareness, missing Acceptance.
- Delete the `test_compile_plan.py` tests that cover only the deleted diagnostics.

### 1c. Demote sealed acceptance to opt-in

- ultraplan stops auto-dispatching the seal-author at invocation. Sealing runs only
  when the operator explicitly asks (e.g. "seal this plan").
- `**Acceptance:** suite` becomes the default for every plan; the collect step is
  skipped when no seal was requested.
- The sealed machinery (`collect_seal.py`, `run_acceptance.sh` sealed mode, the
  vault, `seal_hash.py`, the seal-author agent) stays in the tree, frozen and
  dormant, with its behavioral tests green. Nothing sealed is deleted or rebuilt —
  the instability came from changing this subsystem, and the freeze covers it.
- The gate driver's `sealed` disposition path is untouched; it simply stops being
  reached by default.

### 1d. Declare the freeze

Add to CLAUDE.md's conventions: the verification periphery (sealing subsystem,
gate scripts, run lock, compiler diagnostics) is **frozen** — a change there
requires an eval-measured regression to justify, not an incident narrative.

### 1e. Release mechanics

Bump both `plugin.json` and `marketplace.json` to `0.1.0`. Release commit
`chore(release): 0.1.0 — …` on `main`, tagged only after the Part-2 gate passes.

### Out of scope for 0.1.0

- The git-native seal rebuild (0.2 candidate, only if measured or real use demands it).
- Any change to `waves.js`, `gate_check.py`, `ultra_gate.py`, `run_lock.sh`,
  `run_acceptance.sh`, or the trust checks.
- Any change to the viewer, ultralearn, or ultradocket.

Expected net effect: roughly −1,500–2,000 lines; suite shrinks from ~610 to ~500
tests; no intended runtime behavior change except the simpler default acceptance path.

## Part 2 — The A/B eval (the tag gate)

### Workload

Three benchmark plans, each run once per engine (6 ultrapowers runs total):

1. `evals/fixtures/wide` — existing synthetic; exercises parallel fan-out.
2. `evals/fixtures/chained` — existing synthetic; exercises dependency sequencing.
3. `evals/fixtures/webapp` — **new**: a small real web-app plan (5–7 implementation
   tasks) with its own starter repo, in the operator's target domain. This fixture
   is where code-quality differences can actually appear; the synthetics measure
   engine mechanics.

### Engines

- **A:** ultrapowers 0.0.36 at its release commit (`8a030f4`).
- **B:** the `release/0.1.0` candidate branch.

Each run installs the plugin at its pinned sha into a local marketplace from a git
worktree — the `engine_at_epoch` pattern from the removed eval harness (recover it
from git history; resurrect only the minimal install step, not the harness).

### Run mechanics

- Runs are driven headlessly by `claude -p` from a fresh clone of the fixture repo,
  one at a time (concurrent ultrapowers runs corrupt each other's checkouts).
- **Known risk:** `claude -p` must expose the Workflow tool for a full run. The eval
  kit's smoke test verifies this with the `ultrapowers-probe` workflow before any
  benchmark run; if headless lacks the tool, the runner falls back to driving each
  run in a scripted interactive session, and the results doc records which mode ran.
- A crashed or wedged run is recorded as a failed cell with its transcript kept.
  Reruns are allowed but always recorded as reruns — the results doc must not become
  best-of-N silently.

### Metrics (the three value propositions)

| Prop | Measurement |
|---|---|
| Token economy | Output tokens from the run transcript (the `audit_run.py` harvest method) |
| Clock speed | Wall-clock from launch to gate verdict |
| Code quality | Mechanical counters — gate verdict, suite pass on the integrated tree, redirect rounds, false-block incidents — plus the blind judge |

### The blind judge

One `claude -p` invocation per benchmark plan. Input: the plan text plus both
integrated diffs, labeled A/B with the assignment randomized per plan and the key
withheld from the judge. Output: a defect list per diff and a preference-or-tie
with reasons. The key joins to the results only after judging.

### Tag rule (fixed before any numbers exist)

Tag 0.1.0 when all three hold, comparing B (candidate) to A (0.0.36):

1. **Quality not worse:** the judge finds no defect class present only in B, and
   B's counters (gate verdict, suite pass, redirect rounds, false-blocks) are equal
   or better on every plan.
2. **Tokens equal or better** on every plan, within noise (n=1 per cell — treat
   differences under ~10% as noise).
3. **Clock equal or better** on every plan, within the same noise band.

Any miss → diagnose the regression, fix, re-run only the affected cells (recorded
as reruns), re-check. Never rationalize a miss into a pass.

### Honesty constraints on the eval itself

n=1 per cell makes results directional, not statistical; the results doc says so
plainly. The eval kit is one runner script plus a judge prompt plus fixtures — no CI
wiring, run by hand, one smoke test only. It is permanent (rerunnable before any
future tag) but must not regrow the removed overnight harness.

## Sequencing

1. **Build the eval kit on `main`** — runner, webapp fixture, judge prompt. Touches
   nothing the eval will measure.
2. **Baseline A** — three runs against pinned 0.0.36; commit results immediately.
3. **Cut on `release/0.1.0`** — Part 1, suite green at its new size.
4. **Candidate B runs + judge** — three runs, three judge calls.
5. **Apply the tag rule** — pass: merge, bump manifests, tag, commit the results doc
   as the release's evidence. Fail: the regression is the next work item.
