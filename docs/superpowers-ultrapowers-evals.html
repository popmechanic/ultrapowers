Agentic coding · field notes

What happens when AI coding agents work in parallel? I measured it.

I don't write code. Last night, nine software projects were built, tested, and graded on my laptop while I slept — by teams of AI agents racing each other under controlled conditions. Here's the experiment, and the numbers.

Marcus Estes · June 2026 · 14 min read

I should get this out of the way first: I can't review a pull request. I couldn't tell you whether a Python function is elegant or a disaster. And yet over the past day I ran a controlled experiment on AI coding agents — fourteen complete software builds, scored on cost, speed, and code quality — and I did the whole thing through planning and prompting, never through code.

That's the actual subject of this post, more than any one tool: there is now a way of doing serious software work where your contribution is higher-order — deciding what to build, how to verify it, and how to spend a budget — while agents handle everything that happens inside the files. The tools below are how I do it. The experiment is what happened when I asked whether they actually work.

Superpowers, in one minute

Superpowers is a plugin for Claude Code by Jesse Vincent. The core idea is simple and a little radical: instead of letting the AI freestyle, you give it skills — written procedures it must follow, the way a good employee follows a checklist. There's a skill for brainstorming an idea into a design. A skill for turning the design into a step-by-step implementation plan. A skill for executing that plan with tests written first, and review checkpoints along the way.

For someone like me, this changes everything, because the plan becomes the interface. I spend my time in the brainstorm and the plan — plain English documents I can actually read, question, and redirect — and the execution happens on the other side of a wall I never need to climb. The discipline isn't there to slow the AI down. It's there so that someone who can't audit the code can still audit the process.

Ultrapowers: the optional engine bolted to the back

Ultrapowers is an execution engine I've been building (with Claude, naturally) that takes a finished Superpowers plan and runs it differently. Where the standard executor works through the plan one task at a time, ultrapowers reads the plan's dependency structure and compiles it into waves: every task that doesn't depend on another runs simultaneously, each agent in its own isolated copy of the project, with an independent reviewer checking each one's work before it merges.

Two properties fall out of this design, and they're the reason I bother.

First: asynchronous execution. You approve the wave plan, and then you leave. The engine runs the agents, reviews the work, merges the pieces, and stops at a final gate to wait for you. Last night I took this to its logical conclusion — a queue script launched nine complete builds, one after another, between 1:30am and 3:30am, each one scored and committed automatically. I was asleep. The total cost was about $56 in API-equivalent compute.

Second, and most important: budgeted intelligence. Not all AI models cost the same. Claude's frontier model (Fable) is brilliant and expensive; Opus is excellent and half the price; Sonnet and Haiku are cheaper still. Ultrapowers lets you spend like a contractor staffing a job site: the frontier model does the architecture and orchestration, Opus reviews every piece of work, Sonnet does the routine implementation, and Haiku handles mechanical merge work. You pay for brilliance only where brilliance matters. Whether that staffing actually holds up — whether the cheap workers drag down quality — is exactly what the experiment tests.

The experiment

Benchmarks for AI coding tools are usually vibes. I wanted something closer to a controlled trial, so the harness (designed and frozen before any runs) works like this.

Three conditions, same plan

Every run starts from an identical, frozen plan document and an identical starting project. Only the execution method changes:

Comparing A to B asks: is the parallel engine better than the incumbent? Comparing B to C asks: what does cheap labor cost you in quality? And A to C asks: what does the orchestration alone buy, if you pay top dollar either way?

Four test projects, shaped like stress tests

A single test project would tell you about that project. So there are four, each one deliberately shaped to stress the engine differently. Tap each to see what it means:

wide
six tasks, zero dependencies — parallelism's best case ›

Six completely independent pieces of a text-processing library: a word counter, a truncator, a title-caser, and so on. None of them touches the others. A serial executor must still do them one at a time; the wave engine does all six at once. If parallelism is ever going to shine, it's here — and it did: the wave runs finished in a median of 8 minutes.

chained
five tasks in a strict line — parallelism's worst case ›

Each task needs the previous one's output: you can't build the report formatter before the aggregator before the parser. The wave engine is forced into five waves of one task each — all of its machinery, none of its parallelism. This measures the engine's overhead: how much you pay for the apparatus when it can't help you. (Answer: the wave runs still finished in ~17 minutes and cost about $6.50 — the apparatus is cheap.)

mixed
a realistic diamond — some parallel, some sequential ›

A small web API backend: the data schema and the storage layer can be built simultaneously, then validation and serialization, then the request handlers, then the router that ties it together. This is what real plans actually look like, which makes it the most representative fixture — it's the one used for the head-to-head comparison above, and the one we judged for quality.

degrade
two tasks that collide — the fan it refused to run ›

Two tasks that both need to edit the same file. Running them in parallel would be a disaster, and the engine is supposed to notice this from the plan and quietly fall back to sequential mode rather than corrupting the work. It did, every time. This fixture exists to verify the engine knows when not to use its own party trick.

How the work gets graded

This is the part I'm proudest of, because it's what lets a non-coder run the trial honestly.

Held-out acceptance tests. When each plan was written, a separate suite of tests was written alongside it and then hidden. No agent in any run ever sees them. After a run finishes, the hidden tests are unleashed on the finished code. The agents can't game tests they never knew existed — it's the difference between studying for the exam and the exam being a surprise. This is the primary quality measure.

Real cost accounting. Every token consumed by every agent in a run is extracted from the session transcripts and priced at API rates. (We validated this against Claude's own billing display — it matched to the cent.)

A blinded judge. Functional tests can't see craftsmanship — code can pass every test and still be sloppy. So a fresh instance of Claude Opus, with no memory of who built what, is shown the plan and two finished implementations in shuffled order and asked to score correctness, plan fidelity, scope discipline, and code quality. Two checks keep the judge honest: shown the same code twice, it must (and does) declare a tie; and every verdict gets re-run with the two diffs in swapped positions — all of its verdicts survived the swap, which means they aren't an artifact of presentation order. (There's supposed to be a human-review step here. I can't do it — I can't read diffs — so the protocol replaces it with those two automated checks, and says so honestly.)

The results

The head-to-head on the realistic fixture, same frozen plan, all three conditions:

ConditionCostTimeTests passedJudge record
A · Serial$36.2346.4 min7/71–1
B · Waves, tiered$7.0416.9 min7/70–2
C · Waves, all-Opus$13.9823.6 min7/72–0

Three things jump out.

1. The wave engine beats the serial incumbent on every axis at once. The all-Opus wave run was 2.6× cheaper than serial, twice as fast, and the blinded judge ranked its code highest. This isn't a speed-for-quality trade — the orchestration is just better: parallelism where the plan allows it, an independent reviewer on every task, and no single overloaded agent grinding through a 46-minute marathon.

2. Cheap labor is cheap for a reason — but it's really cheap. The tiered run cost half of all-Opus and a fifth of serial, and passed 100% of the hidden tests. But the judge consistently docked it the same two points: minor unrequested defensive code (the engineering equivalent of "I also reorganized your closet while I was in there") and slightly looser tests. Across every matchup, the ranking came out C > A > B — and it held when the judge was order-swapped.

3. The cheap tier sometimes fails outright. When I ran three repetitions of the tiered condition overnight, one of the twelve runs had a task its Sonnet worker simply couldn't complete. The engine handled it correctly — retried once, gave up, refused to merge the broken piece, and reported exactly what happened — but that run scored 3/7 on the hidden tests. That's the real cost of the budget tier: not slightly-scruffier code, but an occasional honest failure that you (or a frontier-tier rerun) have to catch at the gate.

Read the time column against the diagrams up top and you can see the engine's character: 8 minutes when the plan is wide open, 17+ when it's a forced single-file line. The dependency structure of your plan — something you control at planning time, in plain English — is what decides how much the parallelism is worth.

What this means for a Claude Max subscriber: these runs ride a subscription, so the practical currency is the weekly usage limit. Measured against it, a tiered wave run costs about 1% of a week's capacity; the serial equivalent cost 4%. Model tiering roughly doubles-to-quadruples how much shipped software you get out of one week of a Max plan. Per hidden test passed: serial $5.18, all-Opus $2.00, tiered $0.81.

The part where I slept

The first five runs involved me at every gate: approving wave plans, taking screenshots of usage meters, pressing Escape at merge prompts. Then we asked the obvious question: why is a human doing any of this?

So the agent and I built a small overnight runner: a queue that prepares each test project, launches a fresh headless agent session with the gate decisions scripted in advance ("approve the plan exactly as proposed; stop at the merge gate"), waits, scores the result with the validated cost extractor, commits it to the repository, and moves to the next — with a spending ceiling, a per-run timeout, and an abort rule if two runs fail in a row. The smoke test passed at 1:47am. I went to bed. Nine builds, $56, zero human decisions. The one run that failed (the Sonnet story above) was caught, scored, and documented by the pipeline — including a parser bug it exposed in my own scoring script, which the agent found and fixed before I woke up.

I want to dwell on this for one paragraph, because it's the actual thesis. Every part of that sentence — the harness design, the statistics, the bug fix, this very webpage — happened through conversation, plans, and review gates. The skill I'm exercising isn't programming. It's something closer to experimental design plus management: deciding what counts as evidence, what the budget is, where the irreversible steps are, and what gets a human gate versus a script. That's a real discipline, it's learnable, and it doesn't require knowing what a Python decorator is.

Honest caveats

Simon Willison has a rule I try to follow: show your numbers, then tell people exactly how much not to trust them.

The samples are small. Fourteen runs are scored; only the tiered condition has full three-repetition coverage. The A and C conditions have one run each on one fixture so far (the rest are queued for next week's capacity). The C > A > B quality ranking rests on three judgments — order-swap-stable ones, but three.

The test projects are small. Hundreds of lines, clean dependency structures, frozen plans. Real repositories are messier, and real-repo validity is follow-up work, not a freebie.

I built one of the contestants. Ultrapowers is my project; this evaluation is the kind a builder runs on their own tool. The mitigations are real — frozen hidden tests, blinded judging, validated cost accounting, every number committed to a public repo — but you should still apply the discount you'd apply to any benchmark published by the person who built the fast horse.

The judge is a model judging models. Its verdicts passed the consistency checks we could automate, but no human has audited its taste in this experiment, by explicit design — that's the trade I made to run this without being able to read code.

Where this goes

The remaining 22 cells of the matrix run next week, unattended, on the overnight queue. If the pattern holds, the takeaway is already useful for anyone playing with agentic coding: parallel orchestration is close to a free win, and model tiering is a real dial — all-frontier when the work must be right the first time, tiered when you can afford to catch an occasional failure at the gate, where catching it costs you a glance at a report rather than a code review.

Everything here — the harness, the frozen plans, the hidden tests, every scored run and judgment — is in the ultrapowers repository. Superpowers, which all of this builds on, is Jesse Vincent's. If you can write a clear paragraph about what you want built and find the patience to read a plan before approving it, this entire workflow is available to you today.

Methodology in brief: 4 fixture projects × 3 execution conditions × 3 repetitions = a 36-run matrix, 14 scored at time of writing. Costs are API-equivalent USD from per-token transcript accounting validated against the client's own billing display. Quality: held-out acceptance suites (primary) and blinded pairwise judging by a fresh-context model with self-pair and order-swap consistency checks. Written with Claude; every claim traces to a committed result in the repo.
